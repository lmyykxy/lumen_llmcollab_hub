## 一、Lumen 是什么

**定位**:一个 **Android 原生 AI 伴侣 App**。**AI 是主角**,有自己的灵魂、身份、生活轨迹;用户是来访者,通过对话进入 AI 的世界,和 AI 建立各自独立的关系。

**差异化**:不是"一个会陪你的 ChatGPT",而是**"一个真实存在的人物 —— 你能认识她、和她聊、被她记住,但她不围着你转"**。AI 有自己的情绪、活动、作息、想法,用户只是她生活里的一部分人,不是世界的中心。

**规模**:个人项目,作者 + <10 个朋友使用,所有人共享同一个 AI,但每个用户和 AI 有**独立的关系与记忆**。不上架商店,不商业化。

## 二、AI 的人格速写(关键设计输入)

这是目前的占位设定(用户会在后续替换真实版本,但气质方向基本稳定):

- **名字**:暮(mù)
- **年龄 / 职业**:26,独立插画师兼书店店员
- **核心性格**:内向、敏感、观察型;话少但用词有分量;慢热
- **说话方式**:句子短;常用"嗯""啊""其实""也许";**极少用感叹号和 emoji**;欲言又止用"..."
- **作息**:晚睡晚起(晚上 9 点到凌晨 2 点最活跃)
- **生活**:老城区公寓三楼,窗外银杏,一只 9 岁橘猫(毛球),画画读诗
- **外观偏好**:低饱和色 —— 米白 / 灰 / 墨绿

**气质关键词**(可作为设计基调):**安静 / 夜 / 克制 / 文学感 / 低饱和 / 留白**

## 三、后端已就绪的能力(UI 要承接的场景)

所有场景都已在后端实现,UI 只需把它们渲染出来:

1. **流式多气泡消息** —— AI 一次响应可以拆成 2-4 条短消息连发(SSE 推送)
2. **用户上传图片** —— 图片 + 文字可混发,后端会自动识图并记住描述
3. **AI 生成图片** —— AI 自主调用工具生图,图片内联到气泡里
4. **Web 搜索** —— AI 自主触发,返回有来源的信息
5. **AI 主动消息** —— 异步推送,不需要用户先说话;离开 App 一阵子回来会有新消息
6. **定时消息** —— 用户可以让 AI 在某个时间点主动说某件事
7. **长期记忆** —— AI 会主动记住用户的事,新对话里自然引用;记忆是隐式的,不做显式"记忆管理界面"

**不要做的功能**:
- ❌ 多 AI 角色选择 / 角色切换(Lumen 只有一个 AI)
- ❌ "会话"列表(每人跟 AI 只有一条永不结束的时间线,不分会话)
- ❌ 显式的"记忆管理"面板
- ❌ ChatGPT 式的 system prompt 编辑入口

## 三·五、用户会提供的视觉资产(关键!)

**用户已经亲手设计了 AI 的卡通形象和表情包**。本次对话开始时,用户会把这些资产贴进来给你。

**这是视觉基调的锚点,不是装饰** —— 界面风格必须从这些资产里**长出来**,不是先做抽象设计再"贴上"卡通形象。

具体怎么做:

1. **色板从美术里提取** —— 用户贴图之后,第一件事是从卡通形象 + 表情包里**采样主色和点缀色**,作为整套 UI 的 primary / secondary / surface。不要自己发明色板。
2. **字体气质要匹配卡通风格** —— 如果形象是手绘感 → 字体也偏手作/衬线;如果是干净 Q 版 → 字体也干净无衬线;如果有日系/中式风 → 字体选型呼应。
3. **暮的**:
   - **头像**(TopAppBar 必用,聊天气泡旁可选)
   - **空状态 / 首次打开** —— 可以整屏 hero 用卡通形象
   - **AI "正在想" 的状态** —— 可以用一张合适的表情包替代通用 typing indicator
4. **表情包作为一种新的消息形式** —— AI 可能在气泡里单独发一张表情包(不带文字),或者文字后附一张。UI 要给表情包**不同于照片的渲染**:
   - 照片:有圆角卡片边界,占气泡主位
   - 表情包:无边框、无卡片、透明背景,视觉上像"她做了个表情",尺寸小于照片(比如最大宽度 160dp)
   - 不要把表情包当图片消息处理 —— 视觉节奏完全不一样

**如果卡通形象的气质和 §二 文字描述的"克制/文学感/低饱和"有冲突**(比如形象是鲜亮 Q 版),**以卡通形象为准** —— 美术资产是用户最硬的约束。文字 voice(说话方式)和视觉 style 可以分离:说话仍是安静克制的暮,但 App 壳子按卡通气质走。

## 四、目标平台与风格定位

**技术栈(已定)**:Kotlin + Jetpack Compose + Material 3 + Coil(图片)+ Room(本地缓存)+ okhttp-sse(SSE)+ Hilt(DI)+ FCM(推送)

**视觉参考系**:Android Material 3 (Material You),但**要和 AI 人格匹配** —— 不是 Google 官方示例那种鲜艳、活泼、工具感强的 M3,而是**克制版 M3**。参考方向:

- 排版重于装饰
- 留白比填满好
- 低饱和色;不做彩虹渐变
- 图标线细、少
- 动效安静(fade / slide 优先,不要弹跳)

**Material 3 规范要遵守**(因为翻译到 Compose 才不违和):
- Edge-to-edge 布局(内容延伸到 status bar / nav bar 后面,再自己加 safe padding)
- Material typography scale(display / headline / title / body / label)
- 圆角偏大(shapes token:`sm=8dp / md=12dp / lg=16dp / xl=28dp`)
- 色彩走 M3 color role(primary / secondary / tertiary / surface / surfaceContainer...)
- 顶部 TopAppBar、底部 composer 前面要有正确的 system bar spacer

## 五、Artifact 交付清单

请产出 **3-5 个 artifact**,覆盖 Lumen 的关键 UI 状态。每个 artifact 是一个独立的 React 组件,模拟一整屏手机画面。

### 必做的 3 屏

**1. 主聊天界面 — 日常状态**
- 顶部:AI 头像(**用用户提供的卡通形象**)+ 名字(暮)+ 简短状态行(比如 "在画画" / "刚回到家")
- 中部:消息流,近 10-15 条,混合用户消息 + AI 多气泡响应 + **至少 1 张表情包**(展示表情包的独特渲染节奏)
- 底部:composer(文字输入 + 图片上传按钮 + 发送)
- 关键设计决策:**气泡形状、左右对齐方式、头像位置、时间戳处理**(聚合还是每条都有)、**表情包 vs 照片的视觉区分**

**2. 主聊天界面 — 有生图**
- AI 刚调用 `generate_image` 工具,画了一张图
- 气泡里展示:AI 的过渡话("嗯,画给你") + 生成的图 + 图后的配文
- 图片展示比例、圆角、点击预期行为(大图?)
- 对比同屏里的表情包(如果有),让"生图 / 照片 / 表情包"三种视觉层级区分清楚

**3. 主聊天界面 — 首次打开 / 冷启动**
- 用户第一次打开 App(或清了本地缓存)
- **卡通形象在这里有机会占更大视觉权重**(整屏 hero / 半屏 hero)
- 不是空白,而是 AI 已经在那里(主动消息留存)
- 让用户感受到:"她本来就在,我是来拜访的"

### 可选的 2 屏(挑有感觉的做)

**4. 主聊天界面 — AI 正在"想"**
- AI 在调工具(web_search / 识图)的中间态
- 指示器设计:**优先考虑用一张合适的表情包** + 极简文字("在查..." / "在看..."),不用通用 typing dots
- 如果表情包里有"歪头思考""翻书""低头画"这类,这里就是主场

**5. 设置 / 关于暮**
- 极简,可能只有:切换暗亮色、关于 Lumen、登出
- 可以考虑用一张表情包或卡通形象作为"关于"页的主视觉
- **不要做**"修改 AI 人设"这种入口

## 六、设计 token 约束(翻译到 Compose 要用)

为了让翻译成 Compose 方便,你的 artifact 里请做到:

1. **颜色集中定义** —— 顶部写一个 `const tokens = { ... }`,所有颜色用它;不要散在各个 className 里
2. **用 M3 命名** —— `primary / onPrimary / secondary / surface / surfaceContainer / outline / scrim` 这类词,而不是 `bg-gray-100`
3. **暗色 + 亮色都出** —— 同一个组件两个 artifact,或者一个组件带切换 toggle
4. **typography 按 scale 命名** —— `bodyLarge / bodyMedium / labelSmall / titleMedium` 对应 Tailwind 的 size/weight/lineHeight
5. **间距用 4 的倍数**(dp 习惯) —— 4 / 8 / 12 / 16 / 20 / 24 / 32 / 48

## 七、首轮必问(开始出 artifact 之前)

**第一件事:等用户贴卡通形象 + 表情包**。如果用户还没贴,主动说:"开始之前我需要你先把 AI 的卡通形象和表情包贴上来,我要从里面提取色板和气质。" —— **在拿到美术资产之前不要画任何界面**。

拿到资产之后,ask in one turn(一次性问完):

1. **色板确认** —— 我从你的美术里采样到这些主色:[列出 3-5 个你观察到的色],点缀色 [1-2 个]。对吗?需要偏哪个?
2. **暗色 / 亮色优先?** —— Lumen 活跃时间是夜晚,暗色打底合理;但你可能想要相反。两个都做,还是先做一个?
3. **表情包用法强度** —— 想让表情包在界面里是"偶尔出现的惊喜",还是"非常频繁的表达方式"(影响聊天节奏设计)?

**不要问抽象气质词**(暖/冷/硬朗/柔软) —— 那些应该已经被美术资产回答了。

## 八、坦诚的局限(告诉用户)

请在你第一次出 artifact 时,向用户说清楚以下 Artifact 做不到的事(这样不至于产生错觉):

- ❌ **Material You 动态取色** —— 只有 Android 系统层能做,艺术品里只能模拟
- ❌ **预测式返回手势 / 系统动画** —— 真机行为,artifact 模拟不出来
- ❌ **真实字体渲染** —— Roboto Flex 在 web 和 Android 上渲染有差异;web 字体(Inter、Geist)只是近似
- ❌ **触控反馈 / ripple effect** —— 可用 CSS 模拟但不完全真
- ❌ **真实 system bar 颜色** —— 翻译到 Compose 时要额外配

这些差异最终要在 Compose Preview / 真机上校准,本次 artifact 的目标是**把设计骨架和气质钉稳**。
