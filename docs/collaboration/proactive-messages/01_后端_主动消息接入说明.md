---

## 1. 是什么

跟 `/chat`(用户先发消息,AI 流式回)对应,**主动消息** = **AI 主动开口**,不需要用户触发。三个内置触发源 + 一个用户可自定义的 schedule:

| 触发源 | 频率 | kind | 说明 |
|---|---|---|---|
| `idle_scan` | 每 15 min | `idle_checkin` | 找空闲用户(超过 X 小时没互动),AI 关怀一句 |
| `morning_greet` | 每天 09:00 本地 cron | `morning_greeting` | 早晨给每个 user 一句招呼 |
| `fire_due_schedules` | 每 60 sec | `scheduled` | 扫 `scheduled_messages` 表,fire 已到期的 — 由 AI 在 chat 里通过 `schedule_message` 工具自己排的(例如用户说"30 秒后提醒我",AI 调工具排,到期触发) |

**设计原则**:
- 主动消息**不带 tools**(避免链式触发,AI 自主开口而已,不让它同时排新 schedule)
- DB(`messages` 表)是 source of truth — SSE 推丢了客户端也能通过 `GET /messages?since_id=X` 补拉拿到

---

## 2. 协议:`GET /users/{user_id}/subscribe`

长连 SSE。客户端进前台就连,后台断开。响应是 `text/event-stream`(只要不出错不会自然结束)。

### 2.1 事件类型

| event | data 形状 | 含义 | 客户端动作 |
|---|---|---|---|
| `hello` | `{"user_id": 1}` | 连接建立后第一帧 | 标记"已连上",可以开始接收 |
| `proactive_bubble` | `{"id": 142, "content": "嗯,你今晚还在...", "kind": "idle_checkin"}` | AI 主动发的一个气泡 | 立即渲染。**id 跟 messages.id 一致**,可以用来去重 / lastMessageId 追踪 |
| `proactive_done` | `{"kind": "idle_checkin"}` | 一轮主动消息结束 | 通常 UI 不需要显式动作。如果有"AI 在打字..."状态可以收起 |
| `(SSE comment)` | `:keepalive` | **每 25 秒** 一次心跳 | 客户端 SSE 解析器会自动忽略(SSE comment 行以 `:` 开头),不需要单独处理 |
| `error` | `{"message": "..."}` | 连接侧出错(罕见) | 触发重连 |

**注意 keepalive 周期 25s** — 我们故意比常见反代默认(nginx 60s)短,避免被中间代理误关空闲连接。

### 2.2 一轮主动消息的事件序列(典型)

```
event: hello
data: {"user_id": 20}

(等待…idle_scan 触发或 fire_due_schedules 触发)

event: proactive_bubble
data: {"id": 142, "content": "嗯,你今晚还在?", "kind": "idle_checkin"}

event: proactive_bubble
data: {"id": 143, "content": "我刚才差点睡着。", "kind": "idle_checkin"}

event: proactive_done
data: {"kind": "idle_checkin"}

(继续等下次触发)
:keepalive
:keepalive
...
```

一次"主动开口"通常 1-3 个 bubble(按 `\n\n` 切),然后 `proactive_done`。

### 2.3 跟 `/chat` SSE 的区别

| | `/chat` | `/subscribe` |
|---|---|---|
| 谁先开口 | 用户 | AI |
| 连接形态 | 一次性请求,SSE done 后关 | **长连**,只要不出错不结束 |
| 触发时机 | 客户端 POST 时 | 后端 scheduler / 用户排的 schedule 到期 |
| 工具调用 | 有(generate_image / web_search / schedule_message 等) | **没有**(纯文字主动开口) |
| 事件类型 | delta / bubble_end / tool_call / tool_result / image / done / error | hello / proactive_bubble / proactive_done / :keepalive |

## 3. 客户端接入(给安卓 LLM)

### 3.1 生命周期

```
ON_START  (前台)  → 建立 /subscribe SSE 连接
ON_STOP   (后台)  → 关闭连接

onFailure / onClosed → 退避重连
                       → 重连后立刻 GET /messages?since_id=<本地最大 id> 补拉
                       → 把补拉的 message upsert 到 Room
```

**重要**:重连成功后要先 **GET /messages?since_id=** 补拉,把断开期间错过的 proactive_bubble 拿到。这是因为 backend 推送丢了不会自动重发(进程内 queue,client 不在线就丢)。GET 是 source of truth,能补回来。

### 3.2 退避重连

```
首次失败:1s 后重连
再失败:2s → 4s → 8s → 16s → 30s(封顶)
连接成功后:重置回 1s
```

okhttp-sse 默认有重连机制但策略不一定符合需求 — 建议**关掉默认重连,自己实现**(有 lastEventId / 退避控制)。

### 3.3 UI 渲染契约

`proactive_bubble` 收到时:

```kotlin
// 立即 upsert 到 Room
messageDao.upsert(MessageEntity(
    id = evt.id,  // 跟 messages.id 一致
    userId = userId,
    role = "assistant",
    content = evt.content,
    imageUrl = null,           // 主动消息目前不带图
    quoteRefJson = null,
    createdAt = Instant.now(), // 或后端 created_at(如 GET /messages 拉到再 upsert 覆盖)
    status = "completed",      // 主动消息只在 publish 时才发 SSE,意味着已经 status=completed 入库
))

// UI 跟普通 assistant 消息一样渲染(纯文字气泡),用户视角无区别
```

UI 上不需要给"AI 主动发的消息"做特殊标记(产品味道:让用户感觉"她随时可能开口",不要给"system message"的视觉暗示)。

### 3.4 跟 GET /messages 的关系

- `GET /messages` 返回的所有 message 都包括主动消息(它们也写 messages 表)
- `subscribe` 推送只是**实时通知**,数据本身可以从 GET 拉
- 重启 App 后 → GET /messages 拉历史 → 主动消息自动出现(前端不用特别处理)

## 4. 怎么手动测试一次主动消息

### 4.1 让 AI 自己排一个 schedule(最自然)

在 `/chat` 里发:
```
30 秒后给我提醒一下
```

AI 会调 `schedule_message` 工具排一条 30 秒后的 schedule。前端打开 `/subscribe`,30 秒后应该收到 `proactive_bubble`(kind=`scheduled`)。

### 4.2 服务端直接 publish(测试用)

我可以在后端 shell 里手动调:
```python
await proactive_turn(db, user_id=20, kind="idle_checkin",
                    context="测试主动消息触发")
```

绕过 scheduler,立即 publish。前端验收 SSE 接入时我可以这么帮你 trigger。

### 4.3 等 idle_scan / morning_greet 自然触发

- `idle_scan` 每 15 min 跑一次,但只对"超过 X 小时没活动"的 user 触发(避免刷屏)
- `morning_greet` 每天 09:00 本地时间一次

测试时建议用 4.1 或 4.2,不等 cron。

## 5. 客户端 SSE 解析的几个坑(我之前踩过)

详见后端 memory `feedback_streaming_endpoints.md`,关键的 5 条:

1. **SSE comment(`:keepalive`)的处理**:行首是 `:` 的行是 comment,客户端 parser 忽略即可。okhttp-sse 默认会处理。
2. **重连时要先补拉再连 SSE**:顺序反了的话,SSE 推了条新消息,本地没有它之前的消息 → 时间线出现洞。
3. **CancelledError 不是 Exception 子类**:不影响客户端(只是后端实现细节)。
4. **app 断开 SSE 后,后端 finally 子句负责 unsubscribe**:client 不需要发任何"unsubscribe"消息,就关 socket 即可。
5. **进程内 pub/sub 不持久化**:消息推丢就没了,**只能靠 GET /messages 补拉**。这就是为什么强调"重连后补拉"。
