## 1. TL;DR

联调期间用户在 testbed user_id=20 实际操作时暴露了一系列 hallucination /
工程边界问题。这一波改进是一组连贯修复 — 从观测性到协议层再到 LLM 行为
兜底。前端协议层影响在 §6,**前端不需要改任何代码**。

## 2. 关键改进(按重要度)

### 2.1 观测性(structured logging)

加 trace_id 贯穿一次 chat 全生命周期,SSE 响应头 `X-Trace-Id` 暴露给客户端。
所有 log 走 `event=<name>` 前缀的 logfmt 风格,便于以后接 ELK / Loki。

排查体验:你贴 X-Trace-Id 给我,我直接 `grep "trace=<id>"` 看完整 lifecycle —
LLM input messages dump、tool args、tool result body、reasoning、finish_reason
全在。

日志路径 `~/companion/backend/logs/app.log`(RotatingFileHandler,10MB × 5)。

### 2.2 Vision 切到 Kimi K2.6(去掉 OpenAI vision 依赖)

之前用户上传图走 GPT-4o-mini,现在直接走 Kimi K2.6 多模态 vision。栈统一,
国内线路更稳。前端无影响(vision 是 backend 内部链路)。

附带改进:**AI 自己生成的图也跑一次 self-describe**,67 字左右视觉描述存进
`messages.image_description`,后续 turn 让 Kimi 真"看"自己画过的图(不再
只凭自己写的 prompt 推断)。

### 2.3 Orphan 消息清理机制

**问题**:用户发消息后客户端 kill / SSE 中断,user message 已经写 DB 但
assistant 回复没生成。下次 chat 时 history 里有这条 orphan,Kimi 把它当
"还要处理的请求",触发奇怪行为(画两张图、复刻文字格式、等等)。

**修法**:`messages.status` 字段(streaming/completed/cancelled),`stream_reply`
finally 用 `asyncio.shield + 全新 db session` 批量 update。客户端断开自动
标 cancelled。`get_user_history` 和 `GET /messages` 默认过滤 cancelled —
LLM 看不到 orphan,前端默认看不到。

### 2.4 quote_ref 引用图的视觉信息注入

`quote_ref` snapshot 加 `image_description` 字段(API.md 已同步)。引用带图
消息时 LLM prefix 注入完整视觉描述 + 被引用图 URL。修复"AI 看不到原图无法
img2img"的联调 bug。详见 `06_后端_联调bug修复_引用图视觉注入.md`。

### 2.5 img2img 全链路成熟

- **去 caption**:generate_image 工具不再要求 LLM 给 caption(self-describe
  已经覆盖 LLM 视觉信息需求,caption 反而引入 dedup / 复述 / 跟过渡话碰撞
  等问题)。image message `content=""`,前端按"纯图气泡"渲染
- **assistant_turn.content 修复**:stream_reply 工具循环里 assistant turn
  的 content 字段不再写死 `""`,回灌真实流式输出。修复 Kimi iter=1 误以为
  自己没说过过渡话补一句的 hallucination
- **prompt 写法引导**:`generate_image.md` 拆 "prompt 写作建议" 为
  text2img(完整描述图,100-400 字符)/ img2img(只说变化点,30-100 字符)
  两种风格。修复 Kimi 在"换袜子"等局部修改场景把整张图重描述的问题

### 2.6 OpenAI tool 协议重建(hallucination 复刻治根)

LLM 偶发"用文字伪装工具产物"(reasoning 知道要调工具,content 阶段输出
`[图片: ...]` / `<image-record>` / `[generate_image: ...]` 等格式假装产物)。
这种 bubble 进 DB 后会**自我强化**(LLM 看到自己之前这么写过,继续写)。

双层治理:
- `merge_assistant_bubbles`:assistant 行带 image_url 重建为 OpenAI tool
  协议双 message(`role=assistant tool_calls=[...]` + `role=tool content=...`)。
  LLM 训练里 tool 是另一个角色,不会自我引用复刻
- `finalize_bubble`:detect 伪造 pattern(`[图片:` / `<image-record` 等)
  → 写 `status='cancelled'`,LLM history / GET messages 都看不到

### 2.7 单 turn 同工具最多 1 次

executor 入口 `tools_called_this_turn` set 检查。Kimi 在 tool result 后又
试着调一次 generate_image 时直接拒绝(实测过,用户只发"画一张"模型连画 2 张
的 case)。

### 2.8 错误响应体扁平化

POST /chat 所有错误响应从 FastAPI HTTPException 默认 `{detail: {error,message}}`
嵌套改成扁平 `{code, message}`,适配前端 parser(《02_后端_字段确认回复》
里跟你确认过)。

## 3. 协议层影响(给前端)

| 影响项 | 是否破坏前端 | 处理 |
|---|---|---|
| `messages.content` 字段:image message 现在 `""`(去 caption 后) | ❌ 不破坏 | 前端 4 种组合早已支持"纯图气泡"(content 空 + image_url 非空) |
| SSE `image` 事件 `caption` 字段 | ❌ 保留为 `""`(向后兼容) | 客户端 SSE handler 不用改,值永远空字符串 |
| `messages.status` 新字段(`streaming` 中你**永远看不到**,因为 GET 默认过滤;`completed` 是常态;`cancelled` 也被过滤) | ❌ 不破坏 | 前端可以无视这个字段。如果未来想让用户看"我刚才发的消息但失败了",GET 加 `?include_cancelled=true` 即可,本期没做 |
| SSE 响应头 `X-Trace-Id` | ❌ 新增 | 前端可以记下这个 id,出问题贴给后端,我直接定位 |
| `quote_ref.image_description` 字段 | ❌ DTO 已经 ignoreUnknownKeys | 已经在 hub `docs/collaboration/api/API.md` 反映 |

主项目 `~/companion/docs/API.md` 已同步 `quote_ref.image_description` 字段
跟 hub 权威副本一致。
