## 2. 逐项答复(对应前端 §2)

### §2.1 POST /chat 请求体

前端发的结构是对的:

```json
{
  "user_id": 123,
  "message": "我就笑,怎么了",
  "image_url": null,
  "quote_ref": { "type": "message", "id": "456" }
}
```

| 项 | 答 |
|---|---|
| 字段名 `quote_ref` | ✅ 确认 |
| `type` 全小写 `"message"` | ✅ 确认(`Literal["message"]` 严格小写) |
| `id` 是字符串还是数字 | ✅ **现在两个都接受**(已修)。前端按字符串发即可 |

**修复细节**:`QuoteRefIn.id: int | str`。type=message 时后端用 `int(quote_in.id)` 转 int 后查库;非数字字符串(如 `"abc"`)抛 InvalidQuoteRef → 400 invalid_quote_ref。

### §2.2 GET /messages 响应字段

完全确认前端假设的结构,8 个字段 1:1 一致:

```json
{
  "id": 102,
  "role": "user",
  "content": "我就笑,怎么了",
  "image_url": null,
  "created_at": "2026-04-26T20:01:00Z",
  "quote_ref": {
    "type": "message",
    "id": 456,
    "role": "user",
    "sender_name": "你",
    "preview_text": "你不许笑。",
    "thumbnail_url": null,
    "created_at": "2026-04-26T19:58:00Z",
    "source_event_id": null
  }
}
```

| 项 | 答 |
|---|---|
| 字段名 `sender_name` / `preview_text` / `thumbnail_url` / `created_at` / `source_event_id` | ✅ 全部一致 |
| `role` 必填还是可选 | **后端必填**(`Literal["user","assistant"]`)。前端按 optional 解析 OK,必填字段一定有值 |
| `source_event_id` P0 要前端关心吗 | **不要**,P0 永远 `null`。前端按预留处理(不读)正确 |
| 还有别的字段(`caption` / `image_id`)? | **没有**,8 字段固定 |

⚠️ **响应里 `quote_ref.id` 是 number**(`int`),不是 string。前端按字符串 parse 时记得 JSON number → string 的兼容(Kotlin `JsonElement.jsonPrimitive.content` 之类)。

### §2.3 错误码

| 项 | 答 |
|---|---|
| `409 conversation_busy` | ✅ status code 409 |
| `400 invalid_quote_ref` | ✅ status code 400 |
| 返回体形状 | ✅ **就是 `{ code, message }`**(已修) |

**修复细节**:之前用 `HTTPException(detail={...})` 会被 FastAPI 包成 `{"detail": {...}}`,前端按 `{ code, message }` parse 拿不到。现在改用自定义 `JSONResponse`,返回真正扁平:

```json
{ "code": "invalid_quote_ref",
  "message": "quoted message id=999999 not found" }
```

完整错误码表:

| HTTP | code | 何时 |
|---|---|---|
| 404 | `user_not_found` | user_id 不存在 |
| 400 | `invalid_quote_ref` | quote_ref.id 不存在 / 属于其他 user / 非数字字符串 |
| 400 | `unsupported_quote_type` | 业务层不支持(注:Pydantic 严格 `Literal["message"]` 会先把非 message type 拒成 422,所以这条 P0 实际不会被命中,P1 加多 type 时再启用) |
| 422 | (Pydantic 默认 detail 数组) | 字段缺失 / 类型彻底错(`quote_ref` 不是 object 等) |
| 409 | `conversation_busy` | 同 user 上一条 SSE 还没 done/error |

### §2.4 图片消息引用 preview_text

后端**自动填**:

| 被引用 message | preview_text |
|---|---|
| `content="" + image_url=非空` | `"[图片]"`(自动兜底) |
| `content="随手拍的。" + image_url=任意` | `"随手拍的。"` |
| `content="超过 80 字..." + image_url=任意` | `content[:80] + "…"`(截断省略号) |

前端可以**完全信任后端 preview_text**,不需要本地兜底。

但前端在长按瞬间生成 `pendingQuote`(还没发到后端)时仍需要本地预览逻辑 — 那个保留即可,跟后端最终 snapshot 不冲突,发送时只传 `{type, id}` 后端会重新生成权威 snapshot。

### §2.5 模型上下文注入 ETA

**已交付,跟接收 quote_ref 是同一次部署**(早于本回复)。

实现:`merge_assistant_bubbles` 在拼 LLM history 时,把 user 行的 quote_ref 渲染成 prefix 注入到 user message:

```
[用户在回复 小七:"你不许笑。"]
我就笑,怎么了
```

三种语境的实际注入(测试过):
- 文字引用:`[用户在回复 小七:"你不许笑。"]`
- 图文引用:`[用户在回复 你的一条带图消息:"随手拍的。"]`
- 纯图引用:`[用户在回复 小七的一张图片]`

跟现有 `[用户发了一张图 ...]` 注入对称。**前端 Phase 3 UI 完成后直接联调,不用等模型层**。
