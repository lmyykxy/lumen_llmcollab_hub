## 2. 核心技术决策

### 2.1 PM 拍板的 5 个问题对应实现

| 问 | PM 答 | 落实位置 |
|---|---|---|
| Q1 字段名 message vs content | 保持 `message` | `ChatRequest.message`,文档统一 |
| Q2 sender_name 来源 | A:后端硬编 | `quoting.py::ASSISTANT_DISPLAY_NAME = "小七"` / `USER_DISPLAY_NAME = "你"`,小七真身份替换时改这两行常量 |
| Q3 并发 409 一并做 | 是 | `_BUSY_USERS: set[int]`,同步 check-then-mark,`finally` 兜底 unmark |
| Q4 注入位置 | A:user message prefix | `merge_assistant_bubbles` 在 user 行末追加 image 注释**之后**、role/content 出口**之前**注入 quote prefix |
| Q5 event_log 不做 | 同意 | 完整 quote_ref 存 `messages.quote_ref` JSON,以后 event_log 真做时直接 SELECT 出来 |

### 2.2 注入到模型上下文的具体格式

`merge_assistant_bubbles` 把 user 消息的 quote_ref 渲染成 prefix:

| 引用类型 | prefix 格式 |
|---|---|
| 纯文字 | `[用户在回复 小七:"你不许笑。"]` |
| 图文 | `[用户在回复 你的一条带图消息:"随手拍的。"]` |
| 纯图(content 空) | `[用户在回复 小七的一张图片]` |

跟现有 `[用户发了一张图 ...]`(image_url 的注入)对称,不增加新的 system 段,LLM 看历史 user message 时就能 anchor 引用语境。

prompt 行为约束(PM 文档 §7):
- 模型不会暴露 `quote_ref` 字段名 — 因为注入文本是自然语言,不含技术词
- 模型不应说"根据引用消息" — 这一条目前依赖模型自身行为,**未在 prompt md 加硬规则**。如果实测发现违反,补一条到 `chat_system.md`

## 4. 协议变化摘要(给前端)

简明版,完整在 `docs/API.md`:

### POST /chat 请求
新增可选字段 `quote_ref: {type: "message", id: int}`。

### POST /chat 错误
- `400 invalid_quote_ref`:id 不存在或属于其他 user
- `422`:Pydantic 校验失败(`type` 不在白名单)
- `409 conversation_busy`:同 user 已有进行中的 /chat

### GET /messages 响应
每条 message 新增 `quote_ref` 字段(可能为 null)。结构详见 `API.md` §5。

### SSE 事件
**不变**。引用是 user message 的持久化字段,不需要单独 SSE 帧。

## 5. 已知限制 / 风险

| 风险 | 当前对策 | 何时升级 |
|---|---|---|
| `unsupported_quote_type` 拿到 422 而不是 PM 文档说的 400 | Pydantic Literal["message"] 优先于业务层校验 — 严格 schema 是好事,但跟 PM 文档措辞不一致 | P1 添加 moment/diary/image 时,如果要保留"声明 type 但本版本未实现"的语义,把 Literal 放宽到 str + 业务层校验 |
| 模型可能违反"不说'根据引用消息'"等行为规则 | PM 文档 §7 提到的行为约束目前没在 prompt md 加硬规则,依赖模型自律 | 如果实测发现违反,在 `prompts/chat_system.md` 加一段规则 |
| `_BUSY_USERS` set 是单进程内的 | 多 uvicorn 实例会失效 | 需要横向扩展时换 Redis/PG advisory lock |
| `sender_name` 写死"小七"/"你" | 小七真身份资料到位时改 `quoting.py` 两个常量 | 真身份资料到位 |
| Lock dict 永不清理 | 用户 <10,内存可忽略 | 用户数大时加 LRU |
