## 8. Phase 3：ContextBuilder

当前 `chat_system.md` 只有：

```text
{identity}
当前时间:{now}
{memories_block}
```

这不够。ContextBuilder 应负责统一组装上下文，而不是让聊天代码散装拼 prompt。

### 8.1 输入

```python
@dataclass
class ChatContextInput:
    user_id: str
    user_message: str
    image_url: str | None
    image_description: str | None
    quote_ref: QuoteRefSnapshot | None
    prompt_bundle: PromptBundle
    character_state: CharacterState
    relationship_state: RelationshipState
    recent_messages: list[Message]
    recent_events: list[Event]
    retrieved_memories: list[Memory]
```

### 8.2 上下文预算

```text
core_card:              600-900 tokens
voice_card:             300-500 tokens
safety_card:            200-400 tokens
relationship summary:   200-400 tokens
character_state JSON:   150-350 tokens
retrieved memories:     600-1200 tokens
recent messages:        1000-2500 tokens
recent events:          300-800 tokens
quote_ref / image:      按需注入
```

总量尽量控制在 6k-9k tokens 内。

### 8.3 必须保留的现有行为

```text
1. POST /chat 继续使用 message 字段，不改 content。
2. quote_ref P0 只支持 type=message。
3. quote_ref 继续作为当前 user message prefix 注入模型上下文。
4. 引用图文/纯图消息时，图片描述必须进入 multimodal 或文本上下文。
5. 同 user_id 并发仍返回 409 conversation_busy。
```


```text

引用图文/纯图消息时，模型能看到图片描述。
同 user_id 并发保护不退化。
prompt_version / prompt_sha256 可追踪。
```

### 13.4 图片链路

```text
生图前有 ImageIntent。
固定视觉锚点不漂移。
眼镜按意图可戴可不戴。
generated_images 记录 image_intent / visual_version / prompt_version。
caption 不出现“图片已生成”等工具元话术。
```

## 14. 风险

```text
1. 只替换名字但不换 prompt，会出现“叫小七但还是暮”的人格混杂。
2. 每轮加载完整 prompt 包会导致上下文过长、成本高、回复不稳。
3. TurnAnalyzer 如果同步执行，会拖慢 SSE 体验。
4. CharacterState 如果全靠随机模板，会像假状态，不像连续生活。
5. Relationship 数字如果泄漏到 UI，会破坏产品边界。
6. 历史设计文档仍有“暮”，模型 LLM 可能误读为正式身份。
```
