## 1. Background

用户已补充拍板：

```text
有好感度,但不给前端。
```

这意味着旧边界需要修正为：

```text
内部可以有好感度/关系变量。
前端不能拿到这些变量。
UI 不能出现好感度数字、亲密度数字、关系等级、进度条或恋爱模拟器式玩法。
```

内部关系状态的目的不是做数值玩法，而是让用户从小七的态度、语气、主动程度、分享边界和防御变化中感知关系推进。

---

## 2. Scope

本次 PM 反馈覆盖：

```text
P3 ContextBuilder
P4 CharacterState / RelationshipState DB
P5 TurnAnalyzer
前端可见 API 边界
```

---

## 3. Non-goals

本次不做：

```text
不新增前端好感度 UI
不新增关系等级 UI

不要求 Android 展示任何关系字段
不授权 P6 ImageIntentBuilder 开工
不把 P3/P4/P5 写成最终验收通过
```

---

## 4. PM 判断

### 4.1 P3 ContextBuilder

P3 方向基本成立：

```text
ContextBuilder 收口上下文
预算日志可读
runtime cards forbidden 测试补齐
unknown character fallback 到 xiaoqi
schedule/proactive 工具播报过滤
```

目前未发现需要产品层推翻的问题。

### 4.2 P4 CharacterState / RelationshipState

P4 的内部 DB 方向是对的：

```text
character_states / relationship_states 应保留
trust / intimacy / defense_level 等关系变量可以内部存在
自然语言注入 prompt 的方向正确
```

但 P4 当前 API 暴露需要调整：

```text
GET /users/{id}/character_state 当前返回 relationship_stage。
relationship_stage 对前端来说仍然是关系等级信号。
按用户最新拍板,它不得返回前端。
```

### 4.3 P5 TurnAnalyzer

P5 的方向保留：

```text
TurnAnalyzer 可以继续异步更新 trust / intimacy / defense_level / stage。
内部 delta 可以驱动小七态度变化。
stage 阈值、delta clamp、per-user lock 都符合保守方向。
```

但输出边界必须守住：

```text
模型回复中不能出现“好感度+1”“亲密度提升”“关系阶段升级”等游戏化表达。
前端 response 中不能出现关系变量。
```

---

## 5. Implementation Notes

建议模型/后端先做一个 P4.1 边界修正，再考虑 P6：

```text
1. 调整 CharacterStateResponse。
2. 从 `GET /users/{id}/character_state` 前端响应移除 `relationship_stage`。
3. 保留 DB 内部 relationship_states.stage。
4. ContextBuilder / TurnAnalyzer 内部仍可读取 stage。
5. 增加 API schema 测试,确保以下字段不会进入前端响应：
   affection_score
   trust
   intimacy
   defense_level
   relationship_stage
   prompt_summary
   verifier_report
   hidden_mood
6. 如需要后台调试,另走内部日志或服务端调试渠道,不得复用前端公开接口。
```

推荐前端可见字段：

```text
user_id
mood
current_activity
last_updated_at
```

是否保留 `mood` 仍可由后续体验验证决定；当前它属于生活状态，不是关系数值。

---

## 6. Data / API Changes

### 6.1 内部数据

允许保留：

```text
relationship_states.stage
relationship_states.trust
relationship_states.intimacy
relationship_states.defense_level
内部 affection_score 或等价派生值
```

这些字段只能用于：

```text
ContextBuilder
TurnAnalyzer
主动消息策略
小七回复风格
内部审计/调试
```

### 6.2 前端 API

不得返回：

```text
affection_score
trust
intimacy
defense_level
relationship_stage
stage
prompt_summary
verifier_report
hidden_mood
```

如后端已有公开接口返回这些字段，需视为 P4.1 修正项。

---

## 7. Acceptance Criteria

P4.1 修正完成后，模型/后端回报需包含：

```text
1. `GET /users/{id}/character_state` 最新响应示例。
2. CharacterStateResponse 字段列表。
3. API schema 测试清单。
4. 确认 relationship_stage / trust / intimacy / defense_level / affection_score 不在前端响应。
5. 确认 ContextBuilder / TurnAnalyzer 内部仍能正常使用关系变量。
6. 一次 smoke：同一用户关系变量变化后,小七语气有变化,但前端字段无关系泄漏。
```

---

## 8. Risks

```text
1. 前端拿到 relationship_stage 后自然会做关系 UI,破坏产品边界。
2. 模型直接说出“好感度变化”会让小七变成养成数值对象。
3. 只有内部数值变化、没有表达变化,用户感知不到关系推进。
4. P5 analyzer 过慢或过保守,可能导致态度变化滞后。
```

---

## 9. Next Steps

PM 建议下一步顺序：

```text
1. 模型/后端先做 P4.1 API 边界修正。
2. PM 将 P4.1 与 P3/P4/P5 一起给用户确认。
3. 用户确认后,PM 再写正式验收与 P6 是否开工的授权文档。
```

当前不授权 P6。P6 ImageIntentBuilder 需要等用户确认 P3/P4/P5 与 P4.1 修正方向后再开。
