## 2. 三层 cards 加载策略(PM §7.3 落地)

PM §7.3 给的三层概念:Core / Task / Full。这里的关键设计决策:

```text
Core Card  →  来自 prompts/role/*.md 5 份(P1 写的精简版)
              identity + soul + voice + world + relationships
              拼起来约 9KB / ~2.5K tokens,正好匹配 PM §8.2 给 core 的 600-900 tokens 预算

Task Cards →  来自 prompts/characters/xiaoqi/v0.3.1/prompts/
              visual_card        ← 09_VISUAL_BIBLE.md         (生图任务用)
              image_rules_card   ← 11_IMAGE_PROMPT_GUIDE.md   (生图 prompt 写法)
              diary_card         ← 08_DIARY_MOMENTS_PHOTO.md  (日记/朋友圈/照片)
              lifeflow_card      ← 06_LIFEFLOW_ENGINE.md      (主动消息/state)
              memory_card        ← 07_MEMORY_SYSTEM.md        (P5 TurnAnalyzer 用)

Full Files →  task_dir 下全部 *.md(13 份),只用于离线总结 / 回归测试 / 扩展任务
```

**为什么 Core Card 不直接拼 v0.3.1 6 份核心:**
原始 13 份每份 3-7KB,核心 6 份拼起来 ~25KB / 6000+ tokens,严重超出 PM §8.2 给 core_card 的 600-900 tokens 预算。
P1 写的 `prompts/role/*.md` 已经针对 chat 上下文精简到 ~9KB / ~2500 tokens,正好匹配。
**v0.3.1 的价值在于 Task Cards + 离线参考**,不在于直接当 core。

未来扩展(多角色时):core_card 改成 `prompts/characters/<id>/cards/`,当前单角色项目不需要这一步。

## 3. PromptBundle dataclass

```python
@dataclass(frozen=True)
class PromptBundle:
    character_id:    str             # "xiaoqi"
    version:         str             # "v0.3.1"
    files:           dict[str, str]  # 完整 13 份 v0.3.1 md 文件名 → 内容
    sha256:          str             # 16 字符 hex,内容指纹

    core_card:           str         # 5 份 role md 拼接(每轮 chat 必带)
    voice_card:          str         # 单独抽出 voice(只要语气的 task 用)
    safety_card:         str         # 占位,P3+ 提取专用 safety 段
    relationship_card:   str         # 单独抽出 relationships
    visual_card:         str         # 09_VISUAL_BIBLE.md
    image_rules_card:    str         # 11_IMAGE_PROMPT_GUIDE.md
    diary_card:          str         # 08_DIARY_MOMENTS_PHOTO.md
    lifeflow_card:       str         # 06_LIFEFLOW_ENGINE.md
    memory_card:         str         # 07_MEMORY_SYSTEM.md
```

**sha256 计算口径**:对 `core_items + v0.3.1/* full_files` 全部 (name, content) 排序后哈希。
任何一份 md 改了 → sha 变。日志里出现新 sha 就说明 prompt 包动过,可写入 `model_run_logs` 做回归追溯(P3 接入时使用)。

## 7. 顺手发现 + 修复:proactive finalize_bubble 永远 streaming

测试 6.3 时发现:`scheduler.py::proactive_turn` 写入的主动消息 status 永远是 `streaming`,而不是 `completed`。

**根因**:`save_message(db, user_id, "assistant", clean)` 没传 `status` 参数,默认 `streaming`(为 `/chat` SSE 设计的)。`/chat` 走 `finalize_bubble` 在 finally 里把 `streaming → completed`。但 scheduler 不走 SSE,没人 finalize。

**修复**(`scheduler.py:120` 附近):

```python
# 之前
msg = await save_message(db, user_id, "assistant", clean)

# 之后
msg = await save_message(db, user_id, "assistant", clean, status="completed")
```

Why:proactive 在后台跑,不像 chat 有客户端断开导致的 cancel 场景。写完即定稿。

**清理残留**:测试期间产生的 status=streaming 的 75/76 号主动消息已标 `cancelled`。
