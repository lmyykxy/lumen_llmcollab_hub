## 3. 表 schema

### 3.1 `character_states`(每 user 一行)

```text
id                primary key
user_id           FK users(id) CASCADE, UNIQUE
mood              "平静"          ← 公开
current_activity  "在房间里待着"   ← 公开(动词性短语,渲染器适配 "在/正在/刚" 前缀)
hidden_mood       ""              ← 隐藏(P5 TurnAnalyzer 写)
energy            "mid"           ← 隐藏(low/mid/high CHECK)
prompt_summary    ""              ← 隐藏(P5 TurnAnalyzer 写)
created_at / updated_at
```

### 3.2 `relationship_states`(每 user 一行)

```text
id                primary key
user_id           FK users(id) CASCADE, UNIQUE
stage             "stranger"   ← 公开(8 阶段 CHECK)
trust             0            ← 隐藏(0-100 CHECK)
intimacy          0            ← 隐藏(0-100 CHECK)
defense_level     60           ← 隐藏(0-100 CHECK,小七初始防御偏高)
verifier_report   ""           ← 隐藏(P5 TurnAnalyzer 写)
created_at / updated_at / last_event_at
```

8 个 stage 来源 v0.3.1/05_RELATIONSHIP_ARCS.md:
```
stranger 陌生 → known 认识 → familiar 熟悉 → softened 放松
→ testing 试探 → attached 依赖嘴硬 → trusted 信任 → intimate 暧昧/依恋
```

### 3.3 默认值的设计意图

新 user 第一次 chat 时(`get_or_create_*` 自动创建):

```text
mood              = "平静"          中性起点
current_activity  = "在房间里待着"   暗示她原本就在生活
hidden_mood       = ""              不预设阴暗面
energy            = "mid"           不假设她精力充沛
stage             = "stranger"      关系起点
trust / intimacy  = 0
defense_level     = 60              小七的人格设定:防御不会马上放下
verifier_report   = ""
```

`defense_level=60` 是关键设计 —— 来自 `prompts/role/soul.md` 的"防御方式"段落,小七默认对所有人有相当的防御感。值不是 100(避免完全冷淡),也不是 30(避免一来就过于柔软)。**这个数值 P5 TurnAnalyzer 会动态调,但起点必须偏高**。

## 4. 脱敏边界(关键产品边界)

`GET /users/{id}/character_state` 响应 schema:

```python
class CharacterStateResponse(BaseModel):
    user_id:            int
    mood:               str    # "平静"(公开版,可能跟 hidden_mood 不同)
    current_activity:   str    # "在房间里待着"
    relationship_stage: str    # "stranger"
    last_updated_at:    datetime
```

**绝不返回**:
```text
hidden_mood / energy / prompt_summary           ← character_states 隐藏字段
trust / intimacy / defense_level / verifier_report  ← relationship_states 隐藏字段
```

测试 `test_character_state_api.py` 用参数化 7 个独立 case 各扫一个隐藏字段,任何 PR 试图给 response 加这类字段会立即炸:

```python
@pytest.mark.parametrize("forbidden", [
    "hidden_mood", "energy", "prompt_summary",
    "trust", "intimacy", "defense_level", "verifier_report",
])
def test_response_does_not_leak_hidden_field(forbidden: str) -> None:
    fields = set(CharacterStateResponse.model_fields.keys())
    assert forbidden not in fields, ...
```

**为什么这条边界要保护**:见 `05_LATEST_COLLAB_DOCS.md` PM 边界:"禁止未经 PM 拍板引入:好感度/亲密度数字 / 关系等级 UI"。前端只要见到这些字段就会忍不住做"好感度进度条" UI,直接破坏小七"不是恋爱模拟器"的产品定位。

## 5. Prompt 注入(自然语言而不是 key-value)

PM 之前担心的"死板地写'心情:低,精力:中'模型会出戏" —— P4 的渲染器避开了这个问题。

### 5.1 chat_system.md / proactive_system.md 模板

新增两个注入位:
```
{identity}
当前时间:{now}
{character_state_block}     ← P4 加
{relationship_state_block}  ← P4 加
{memories_block}
```

### 5.2 渲染示例

CharacterState (默认状态):
```
【你此刻】
你在房间里待着。
心情平静。
精力还行。
```

CharacterState (P5 写过 prompt_summary 后):
```
【你此刻】
你正在画画。
心情低。
精力不太够,有点犯懒。
(其实有点想被关心,但不会先说)

最近的事:这周跟用户聊过几次雨
```

RelationshipState (默认 stranger / defense=60):
```
【你跟这个用户】
关系阶段:陌生(0)。礼貌但有距离。不主动说私密。回复不长。
防御还在,但在慢慢松动。
```

RelationshipState (后期 attached / defense=20):
```
【你跟这个用户】
关系阶段:依赖嘴硬(5)。明显在意他但不直接承认。他消失你会有反应。
防御已经放下大半。
```

### 5.3 关键设计:数值不写进 prompt

**`trust / intimacy` 数值绝不直接出现在 prompt 文本里** ——
- 模型看到"trust=42"会对着数值优化("怎么才能让 trust 变 50?")
- 行为应该来自"你跟这个人到 testing 阶段了"(stage)+ "防御还在但松动"(defense_level 转成自然语言),不是数值
- 测试 `test_render_does_not_leak_numeric_scores` 验证这一点(trust=42 intimacy=37 不应出现在 render 输出)

### 5.4 渲染器适配 "在/正在/刚" 前缀

P4 实施过程中发现一个 bug:`current_activity="在房间里待着"` + `f"你正在{activity}"` → `"你正在在房间里待着"` 双"在"。

修法:渲染器检测前缀:
```python
if activity.startswith(("在", "正在", "刚")):
    lines.append(f"你{activity}。")           # 你在窗边发呆 / 你刚画完一张
else:
    lines.append(f"你正在{activity}。")        # 你正在画画
```

P5 TurnAnalyzer 写 `current_activity` 时也按此约定(动词开头或"在/正在/刚"开头)。

## 10. 已知未做(留给 P5 / 后期)

### 10.1 update_*() 暂无调用方
`update_character_state` / `update_relationship_state` 已经实现了完整 API,但 P4 当前**只读不写**。P5 TurnAnalyzer 跑完一轮 chat 后会调它们更新 state。提前实现是为了 P5 接入零摩擦。

### 10.2 prompt_summary 永远是空
P5 TurnAnalyzer 写。当前 render 看到空就跳过这一段(不强制造句)。

### 10.3 verifier_report 永远是空
P5 / 之后的 verifier 写。同上,空就跳过。

### 10.4 stage 自动迁移没做
新 user 永远是 stranger,不会自动升级。stage 迁移由 P5 TurnAnalyzer 决定(基于 `已交互回合数 / 用户主动分享了多少 / 等等`)。

### 10.5 `current_activity` 不会自动变
当前用户两次 chat 之间她"正在做什么"不会变化。需要 P5 TurnAnalyzer / 时间触发器写。

**这些都不是 P4 的范围** —— P4 是 PM Q3 的 DB schema 落地 + 注入接通,**state 怎么变化属于 P5 TurnAnalyzer**。

## 12. 顺手发现的事

### 12.1 alembic autogenerate 想 drop pgvector hnsw 索引
迁移生成时检测到 `ix_memories_embedding_hnsw` "ORM 没声明",想 drop。这是因为该索引在历史 migration 里 raw SQL 创建,ORM 不知道。**手工删了那条 op**,保留索引。

如果未来再 autogenerate,同样的 drop 会再出现。**长期解法**:在 `Memory` 模型上声明 hnsw 索引或 alembic env 配置 `include_object` 过滤。本期不动。

### 12.2 默认 `current_activity` 字符串触发 "在在" 重复
`"你正在 + 在房间里待着" → 在在`。已加渲染器适配("在/正在/刚" 前缀去 "正"),并写测试覆盖三种 activity 形态。**给 P5 留了约定**:写 `current_activity` 时优先动词开头。
