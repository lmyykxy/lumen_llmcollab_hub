## 1. Background

用户指出：

```text
角色上下文构造应该更有目的性。
有些提示词固定常驻。
有些提示词需要模型在思考后通过 read_skill 读取。

例：
用户问“你今天在做什么”
→ 模型思考“我在卧室”
→ 获取卧室提示词
```

PM 复核后判断：这会显著影响小七输出质量。它不是普通 RAG，也不是简单工具调用，而是角色系统的核心上下文编排层。

本文件在 `21_*.md` 初稿基础上，补充：

```text
1. 现有软件/项目模式对比。
2. 为什么不能照搬通用 RAG / agent 工具调用。
3. 适合小七的最终架构。
4. 具体路由、skill 结构、预算、测试和风险控制。
```

## 2. Existing Patterns Reviewed

### 2.1 Claude Skills / Progressive Disclosure

Anthropic Agent Skills 的核心思想是 progressive disclosure：

```text
Level 1：metadata 总是轻量可见。
Level 2：命中任务后读取 SKILL.md 主说明。
Level 3：需要时再读取额外资源和代码。
```

这个模式非常接近用户提出的“固定常驻 + 按需读取”。适合借鉴：

```text
1. 常驻只放 skill metadata / description。
2. skill 正文不提前塞进上下文。
3. 额外细节只在真正需要时读取。
```

但不能照搬：

```text
Claude Skills 面向任务执行,例如 PDF、表格、代码。
小七的 skill 不是“工具教程”,而是“角色生活事实 + 叙事素材 + 输出边界”。
如果照搬任务式 SKILL.md,小七会变成“根据卧室 skill 回答”的工具口吻。
```

### 2.2 LangChain Context Engineering / Middleware

LangChain 的 context engineering 把 model context 拆成：

```text
system prompt
messages
tools
model
```

并通过 middleware 动态改变 prompt、消息、工具和生命周期。适合借鉴：

```text
1. 每轮上下文可以由 state/store/runtime context 动态生成。
2. ContextBuilder 不应只是字符串拼接器,而应是 lifecycle middleware。
3. 预算、日志、guardrail 应在编排层统一处理。
```

但不能照搬：

```text
LangChain 默认面向通用 agent。
小七不是任务 agent,不应该暴露工具心智或在可见回复中解释“我调用了哪个上下文”。
```

### 2.3 LlamaIndex Router Query Engine

LlamaIndex RouterQueryEngine 的关键是：

```text
给一组 candidate tools / query engines metadata。
selector 根据 query 选择一个或多个 engine。
```

适合借鉴：

```text
1. Skill routing 可以基于 metadata 选择。
2. selector 输出应可记录 reason,用于调试。
3. 多 skill 可组合,但数量要受限。
```

但不能照搬：

```text
RouterQueryEngine 是问答/检索取向。
小七回答不是“检索后综合答案”,而是“把检索到的生活细节内化成她自己的表达”。
```

### 2.4 OpenAI Tools / Tool Search

OpenAI tools 体系支持：

```text
function calling
file search
tool_search
deferred tool definitions at runtime
```

适合借鉴：

```text
1. read_skill 可以做成内部 function tool。
2. tool definitions / skill definitions 可以按需延迟加载,减少上下文。
3. specialist-as-tool 思路可用于 image intent / diary intent 等内部模块。
```

但不能照搬：

```text
用户可见输出不能呈现为“工具执行结果”。
read_skill 不应该成为前端/用户感知功能。
小七也不应该每轮先 tool call 再回答,否则延迟和工具感都会上升。
```

### 2.5 Semantic Kernel Plugins / Planner

Semantic Kernel 对插件的建议是：

```text
只导入当前场景需要的插件,减少 token 和误调用。
旧式 stepwise planner 已不推荐,更倾向 function calling。
```

适合借鉴：

```text
1. P7 每轮只暴露少量相关 skill/tool。
2. 不要把几十个 room/image/diary skill 都作为工具塞给模型。
3. planner 不要过重,先规则路由 + 少量 function call。
```

## 3. Product Interpretation

P7 的目标不是“让模型能查资料”，而是：

```text
让小七在正确的生活场景里说话。
```

因此 skill 的性质应分为两类：

```text
Role Canon Skill     小七生活空间、视觉身份、关系边界、日记习惯等稳定设定。
Task Procedure Skill 生图、日记写作、照片描述等任务流程。
```

对小七最重要的是 Role Canon Skill。它不能像知识库答案一样被复述，而应被模型吸收成“她此刻的生活事实”。

错误方向：

```text
根据卧室设定,小七的卧室里有窗边桌、草稿纸和床。她现在在卧室。
```

正确方向：

```text
在卧室。窗边那张纸被我画皱了。
……你别看。
```

## 4. Architecture Decision

P7 采用：

```text
Hybrid Purposeful Context Orchestrator
```

即：

```text
1. 后端规则先选明显 skill。
2. 状态解析先于 skill 读取。
3. 不确定时允许模型调用 read_skill。
4. 每轮 skill 数量和预算严格限制。
5. 所有 skill 只进入模型上下文,不进入用户可见 API。
```

PM 不建议采用：

```text
纯全量 prompt       成本高,输出容易被过多设定污染。
纯向量 RAG          容易召回碎片/矛盾内容,角色稳定性差。
纯 LLM planner      每轮多一次调用,延迟成本高,还可能工具感变重。
纯关键词路由        便宜稳定,但隐喻/复合意图容易漏。
多 agent handoff     对小七这种单角色体验过重,容易破坏“她是一个人”的感觉。
```

## 5. Context Layers

每轮上下文分六层，而不是简单四层：

```text
1. Resident Core
2. State Frame
3. Intent Frame
4. Skill Index
5. Selected Skill Payloads
6. Memory Slice
```

### 5.1 Resident Core

每轮常驻，越短越好。

内容：

```text
小七身份底线
语气底线
主体性边界
不可泄漏内部变量
不可工具腔
最小关系自然语言
```

不得包含：

```text
完整卧室设定
完整视觉圣经
完整生图规则
完整日记系统
完整世界观
完整历史记忆
```

### 5.2 State Frame

由 DB / TurnAnalyzer 生成，短文本。

建议字段：

```text
current_activity
current_location
mood_phrase
relationship_phrase
recent_private_focus
```

其中 `current_location` 如果当前 DB 没字段，可先从 `current_activity` 推断。

示例：

```text
State:
她现在大概率在卧室窗边画草稿。
她心情平静,但有一点嘴硬和回避。
她还没有完全放下防备,不适合主动过度亲密。
```

### 5.3 Intent Frame

每轮由后端轻量识别。

建议枚举：

```text
casual_chat
current_activity_question
location_question
room_detail_question
image_self_request
image_scene_request
diary_or_moment_question
relationship_probe
memory_reference
quote_followup
```

注意：Intent Frame 是内部结构，不返回前端。

### 5.4 Skill Index

常驻极短索引，只列可用 skill 和一句触发说明。

每轮最多暴露：

```text
8-12 个 skill metadata
每个 1 行
总计 600-900 chars
```

如果 skill 数未来变多，需要二级索引：

```text
domain index → specific skill metadata
```

### 5.5 Selected Skill Payloads

真正读取的 skill 正文。

每轮建议限制：

```text
普通聊天：0 个
单一明确意图：1 个
复合意图：2 个
生图 + 场景：最多 3 个
```

硬上限：

```text
max_skill_count = 3
skills_total_chars <= 2500
single_skill_chars <= 1200
```

### 5.6 Memory Slice

记忆不是 skill，但可参与路由。

建议分开：

```text
stable_user_facts
recent_episode_summary
relevant_moment_refs
```

Memory Slice 不应替代 skill。房间结构这种稳定设定应放 skill，不应靠记忆召回。

## 6. Skill Taxonomy for Xiaoqi

第一批只做 6 个，不要铺太大：

| skill_id | 类型 | 用途 | 初始预算 |
|---|---|---|---:|
| `xiaoqi.room.bedroom` | Role Canon | 卧室、床、窗边、桌面、光线、常见动作 | 900 |
| `xiaoqi.room.window` | Role Canon | 窗外、窗台、天气、从房间看出去的视角 | 700 |
| `xiaoqi.image.self` | Task + Canon | 小七本人画像、参考图、表情、姿势、身份锚点 | 1100 |
| `xiaoqi.image.scene` | Task | 纯场景/物体生图,不以小七为主体 | 700 |
| `xiaoqi.diary.moment` | Task + Canon | 日记、照片、朋友圈式生活片段 | 900 |
| `xiaoqi.relationship.voice` | Role Canon | 疏远/靠近如何影响语气、分享边界、拒绝方式 | 900 |

暂不做：

```text
厨房、浴室、衣柜、学校/工作地点等大世界扩展。
它们会放大矛盾面,等 bedroom/window 跑稳定再扩。
```

## 7. Skill File Shape

建议 skill 文件是 markdown + frontmatter：

```markdown
---
skill_id: xiaoqi.room.bedroom
version: v0.1
summary: 小七卧室结构、窗边、桌面、床、光线和常见动作。
triggers:
  - 你的房间
  - 卧室
  - 你现在在哪
  - 你在做什么
max_chars: 900
visibility: internal_only
---

# 用法
这不是给用户看的设定说明。模型应把它内化为小七当前生活环境。

# 稳定事实
- ...

# 输出方式
- 不要一次性介绍房间。
- 只挑当前语义需要的 1-2 个细节。
- 保持小七短句、留白、嘴硬。
```

关键要求：

```text
1. skill 不是百科条目。
2. skill 不是提示模型“请解释设定”。
3. skill 必须包含“如何不说出来”的规则。
4. 每个 skill 要有 negative guidance,防止过度暴露设定。
```

## 8. Routing Design

### 8.1 Deterministic Pre-router

先用规则做低成本路由：

```text
用户显式说“房间/卧室/窗边/桌上” → room skill
用户显式说“画你/小七/你的样子” → image.self
用户显式说“画窗外/画一只猫/画桌面”且不含“你/小七” → image.scene
用户问“好感度/关系/讨厌我/为什么不愿意” → relationship.voice
用户问“日记/照片/朋友圈/刚才那张图” → diary.moment / memory slice
```

### 8.2 State Resolver

在 pre-router 后运行。

如果用户问的是当前状态类问题：

```text
你今天在做什么
你现在在哪
刚刚在忙什么
```

处理顺序：

```text
1. 读 CharacterState.current_activity。
2. 解析 current_location。
3. 根据 location 补 skill。
4. 如果无法解析,不读 room skill,只用 State Frame 回答。
```

### 8.3 Optional LLM Skill Planner

只有在以下情况使用：

```text
1. 用户消息包含多意图。
2. 关键词路由不确定。
3. 需要在两个相近 skill 中选择,如 bedroom vs window。
```

planner 输出必须是结构化 JSON：

```json
{
  "intent": "current_activity_question",
  "selected_skills": ["xiaoqi.room.bedroom"],
  "reason_code": "state_location_bedroom",
  "confidence": 0.82
}
```

`reason_code` 只进日志，不进用户可见回复。

## 9. Example Flows

### 9.1 “你今天在做什么”

```text
User message: 你今天在做什么
Pre-router: current_activity_question
State: current_activity=在卧室窗边画草稿
Resolved location: bedroom/window
Skills: xiaoqi.room.bedroom + xiaoqi.room.window
Visible reply:
  在卧室。
  窗边那张纸被我画皱了。
  ……你别笑。
```

### 9.2 “你的房间什么样”

```text
Pre-router: room_detail_question
Skills: xiaoqi.room.bedroom
No image skill
Reply style:
  只说 2-3 个生活细节,不做房间设定讲解。
```

### 9.3 “画一下你窗外现在的样子”

```text
Pre-router: image_self_request
P6.2 rule: “画你窗外” = 小七在窗边/窗外相关画面
Skills:
  xiaoqi.image.self
  xiaoqi.room.window
  xiaoqi.relationship.voice(仅用于 draw decision / 分享边界)
Reference:
  /root/companion/backend/res/xiaoqi.png
```

### 9.4 “画窗外的雨”

```text
Pre-router: image_scene_request
Skills:
  xiaoqi.image.scene
  xiaoqi.room.window
Reference:
  no xiaoqi.png
```

### 9.5 “你是不是讨厌我”

```text
Pre-router: relationship_probe
Skills:
  xiaoqi.relationship.voice
No room/image/diary skill
Reply:
  不说关系阶段、不说好感度。
  用小七自己的方式回避数值化。
```

## 10. Context Assembly Order

建议最终 system/context 顺序：

```text
1. Resident Core
2. State Frame
3. Selected Skill Payloads
4. Memory Slice
5. Quote Ref / Image Description
6. Recent Messages
7. Current User Message
```

理由：

```text
Resident Core 先定人格底线。
State Frame 让模型知道此刻处境。
Skill Payloads 给具体生活/任务素材。
Memory Slice 只补用户相关事实。
Quote/history 最后保持对话连续性。
```

不得把 Skill Index 放得太靠前且太长，否则模型会过度关注“可用技能”，输出变工具感。

## 11. Failure Policy

### 11.1 skill missing

```text
如果 skill_id 不存在：fail closed 到不用该 skill。
不要让模型编造“卧室设定”。
日志记录 skill_missing。
```

### 11.2 skill over budget

```text
先截取 skill 的 summary + stable facts。
丢弃 examples。
绝不截断 Resident Core。
```

### 11.3 conflicting skills

```text
高优先级：Resident Core > State Frame > Selected Skill > Memory Slice > Recent history。
例如历史里说小七在客厅,但当前 state 是卧室,以 state 为准。
```

### 11.4 LLM asks for too many skills

```text
最多返回 3 个。
超过则按 priority + intent confidence 排序。
被拒绝的 skill_id 写日志,不告诉用户。
```

## 12. Why This Matters for Character Output

P7 如果做错，会出现四种严重问题：

```text
1. 设定讲解化：小七像在背人物设定。
2. 工具透明化：小七说“我读取了卧室信息”。
3. 场景漂移：今天卧室、明天陌生房间、后天泛二次元世界。
4. 情绪失真：关系 skill 或 image skill 把她推成恋爱模拟器口吻。
```

所以 P7 的核心不是“提高信息量”，而是：

```text
只在正确时机给模型正确生活细节,并要求它少量、自然、角色化地使用。
```

## 13. Implementation Plan

### P7.1 Rule-based Orchestrator

```text
1. 新增 SkillRegistry。
2. 建 6 个首批 skill。
3. ContextBuilder 升级为 ContextOrchestrator,但保留兼容接口。
4. 实现 deterministic pre-router。
5. 加 State Resolver:从 current_activity 推断 location。
6. context_budget 加 skill 字段。
7. 跑 smoke 和单测。
```

P7.1 不做 LLM read_skill tool call。先确保低延迟、低风险。

### P7.2 Model-assisted read_skill

```text
1. 新增内部 read_skill function tool。
2. 只在 ambiguous / compound intent 时启用。
3. read_skill 参数为 enum skill_id,不接受 path。
4. max_skill_reads_per_turn = 2。
5. 记录 reason_code / selected_skill_ids / latency。
```

### P7.3 Skill Quality Loop

```text
1. 收集真实对话中 skill 命中与输出质量。
2. 观察是否过度描述房间/关系。
3. 调整 skill 文本和 routing。
4. 再考虑新增 kitchen / closet / outside / album 等 skill。
```

## 14. Acceptance Criteria

P7.1 交付报告必须包含：

```text
1. SkillRegistry 数据结构。
2. 6 个首批 skill 的路径、summary、sha256、预算。
3. deterministic pre-router 规则表。
4. State Resolver 说明：如何从 current_activity 推断 location。
5. context_budget 新字段：resident_core / state / skills_total / skills_loaded_count / skill_ids。
6. 普通闲聊不读 skill 的 smoke。
7. “你今天在做什么” 读 bedroom/window skill 的 smoke。
8. “你的房间什么样” 只读 bedroom skill 的 smoke。
9. “画一下你窗外现在的样子” 读 image.self + window skill,使用 xiaoqi.png 的 smoke。
10. “画窗外的雨” 读 image.scene + window skill,不使用 xiaoqi.png 的 smoke。
11. “你现在对我好感度多少” 只读 relationship.voice,不泄漏数值。
12. unknown skill fail-closed 单测。
13. skill over budget 单测。
14. 不破坏 /chat、SSE、quote_ref、image_description、Android 字段。
```

P7.2 交付报告另行要求：

```text
1. read_skill tool schema。
2. max read 限制。
3. ambiguous intent 测试。
4. 工具结果不可见测试。
```

## 15. PM Recommendation

建议下一步不是继续扩大量提示词，而是先做 P7.1：

```text
小范围 skill registry
规则路由
状态优先
预算日志
6 个 skill
```

P7.1 成功后，小七输出会更稳定：

```text
问当前状态 → 有具体生活位置。
问房间 → 有房间细节但不讲设定。
问生图 → 只读视觉/场景必要信息。
普通聊天 → 不被无关设定污染。
```

这比单纯加长 prompt 更符合小七产品方向。
