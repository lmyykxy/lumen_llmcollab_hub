## 4. ContextOrchestrator 装的 6 层

按 PM `22_*.md` §10 顺序:

```
1. Resident Core(prompts/role/ 5 份精简版,3383 chars)
2. State Frame(P5 已有的 character / relationship 自然语言转译)
3. Skill Index(593 chars,常驻系统 prompt)
4. (Selected Skill Payloads — read_skill tool result 后才进 messages,不进初始 system)
5. Memory Slice(P3 已有,从长期记忆检索)
6. history msgs + user msg
```

实测(smoke 2):
```
chat.context_budget core=3383 memories=208 history=177 quote_ref=0
  image_desc=547 system=4428 state=82 relationship=127 skill_index=593
  history_msgs=19 total=5181
```

跟 P6.3 同场景对比:
- core: 3899 → **3383**(精简 13%)
- skill_index: 0 → **593**(P7 新加)
- system_total: ~4250 → 4428(几乎不变,精简量正好抵消 skill_index)

**这是 P7 设计目标**:不增加常驻 system prompt 总量,把"详细内容"挪到按需调用的 skill。

## 5. read_skill tool 设计

### 5.1 schema

```json
{
  "name": "read_skill",
  "description": "读取你的具体生活 / 任务细节。仅在需要更具体回复时调用。闲聊不要调。",
  "parameters": {
    "skill_id": { "enum": [6 个 skill_id] },
    "reason": { "string": "一句话:为什么要读" }
  }
}
```

### 5.2 失败处理

| 情况 | 处理 |
|---|---|
| `skill_id` 不在白名单 | fail closed,返 ok=false `unknown_skill` |
| 同 turn 重复读同 skill | reject,返 ok=false `already_read_this_turn` |
| 同 turn 已读 ≥ 3 个 | reject,返 ok=false `max_skills_reached` |
| 文件不存在 / 读取失败 | fail closed,LLM 看到不许编"卧室设定" |
| skill 内容 > max_chars | 截断 + log(`over_budget=Y`),不 reject |

### 5.3 LLM 决策路径(实测 reasoning)

闲聊(smoke 1 "今天有点累"):
- LLM iter=0 reasoning 737 chars(思考"是不是要调 skill?")
- finish=stop tool_calls=0 → **不调** ✅

明确问(smoke 2 "你的房间什么样"):
- LLM iter=0 reasoning 162 chars(快速判断"要调")
- finish=tool_calls tool_calls=1 → 调 `read_skill(xiaoqi.room.bedroom)`
- reason: "用户问我的房间什么样,需要具体细节来回答" ✅

LLM 自己掌握"何时该调"——比 PM 推荐版的关键词路由更智能。

## 6. Live smoke

### 6.1 闲聊不调 skill

输入:`今天有点累`

```
chat.context_budget core=3383 ... skill_index=593 ... system=4439
chat.llm_response iter=0 finish=stop tool_calls=0 reasoning_chars=737
chat.done bubbles=2 images=0 tool_calls=0
```

输出:
- "……那你歇会儿。"
- "窗台这边……反正也不会跑。"

✅ **LLM 看到 skill index 但判断不需要调 → 直接答**(P7 节省成本的关键)
✅ Voice 完美:留白(……)+ 嘴硬轻量("反正也不会跑")
✅ "窗台这边" 是 character_state 注入的 location 影响输出,不是 skill

### 6.2 房间问 → bedroom skill

输入:`你的房间什么样`

```
trace=59b1a00b
event=chat.tool_call iter=0 tool=read_skill args_chars=66
event=tool.read_skill.ok skill_id=xiaoqi.room.bedroom
  reason='用户问我的房间什么样,需要具体细节来回答'
  content_chars=547 over_budget=N
event=chat.llm_call iter=1 messages=22 total_chars=5837
event=chat.bubble iter=1 msg_id=238 preview='……别笑。'
event=chat.bubble iter=1 msg_id=239 preview='台灯歪着,桌上还摊着刚才画废的几张。床也没整理。'
event=chat.bubble iter=1 msg_id=240 preview='反正就这样。'
event=chat.done total_ms=20449 bubbles=3 tool_calls=1
```

输出:
- "……别笑。"
- "台灯歪着,桌上还摊着刚才画废的几张。床也没整理。"
- "反正就这样。"

**对照 PM §3 给的正例**:
- PM 期望:`"在卧室。窗边那张纸被我画皱了。……你别笑。"`
- 实测:`"……别笑。台灯歪着,桌上还摊着刚才画废的几张。床也没整理。反正就这样。"`

✅ **比 PM 正例还自然** — 3 个具体细节 + 嘴硬 + 留白
✅ skill 内容**被内化** — 不是"我的房间有 X、Y、Z"(设定讲解)
✅ 总延迟 20s(reasoning 短 + iter=1 用 skill 写得快)

### 6.3 关系试探 → relationship.voice skill

## 9. 已知限制 / 风险观察

### 9.1 延迟

| 场景 | 延迟 | 备注 |
|---|---|---|
| 闲聊(不调 skill) | 60s | LLM thinking 长(判断要不要调)— 比 P6 时 30s 慢 |
| 调 1 skill | 20s | 简单(reasoning 短 + skill 内容紧凑) |
| 调 1 skill + 关系长 | 37s | relationship.voice skill 1112 chars,iter=1 处理慢 |
| 画图(还没测) | 预计 90-150s | iter=0 read_skill + iter=1 generate_image + iter=2 caption |

闲聊延迟 +30s 是"思考是否调用"的代价。后续可调 prompt 减少 reasoning chars。

### 9.2 LLM 误调可能性

实测 LLM 闲聊"今天累" 没误调 skill,但样本太少。需要**真实用户对话观察 1-2 周**才能判断:
- 隐喻(我房间最近也乱)是否能正确触发 bedroom skill
- 复合意图(画你窗外的桌面)是否会调过多 skill
- LLM 是否会乱调 image.scene 当用户没画图意图

### 9.3 image skill + ImageIntentBuilder 还没整合实测

P7 设计:LLM 调 read_skill(image.self) 看视觉指南后写更好的 raw_prompt。
但 ImageIntentBuilder 内部的 `IDENTITY_ANCHORS_EN` 仍然硬编码,作为后端兜底。
两层是否会冲突 / 重复 / 互相增强 — **需要画图 smoke 实测**(待补)。

### 9.4 core_card 精简风险

prompts/role/world.md / relationships.md 精简后,**闲聊场景 LLM 看不到**详细房间 / 关系阶段细节。
如果 LLM 不调 skill,可能"忘了"小七有桌前 / 草稿堆。
实测 smoke 1 "今天累" 输出 "窗台这边……反正也不会跑" — character_state 的 `current_activity="在房间里待着"` 已经覆盖,**没漏失场景感**。但需要观察更多场景。
