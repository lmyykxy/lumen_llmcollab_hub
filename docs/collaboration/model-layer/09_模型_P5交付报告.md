## 3. 用户拍板的设计选择

### 3.1 频率(选 1.B):每 N=3 用户消息一次

实现:`character_states.last_analyzed_message_id` 记录上次分析的 user msg id。
每轮 chat 跑完,数 `id > last_analyzed_message_id` 的新用户消息;< 3 直接 skip。

env:`LUMEN_TURN_ANALYZER_THRESHOLD=3`(默认 3,可调)。

**为什么不每轮跑**:
- 每轮多一次 LLM 调用 → chat 成本翻倍
- 状态本来就该缓变 —— 用户说"嗯"AI 不该立即更新 mood

**为什么用 last_analyzed_message_id 而不是计数器**:
- 计数器需要每轮 INSERT/UPDATE,重启可能丢
- 用 max user msg id 是幂等的:重启后再读还是同一个 cutoff,自然续上

### 3.2 模型(选 2.A):用主 chat 同 provider

实现:`provider = chat_provider()` → 当前是 Kimi K2.6 thinking。
`max_tokens=4000`(给 thinking 留 buffer + JSON 输出 ~500 tokens)。

**为什么不另引入便宜模型**(如 GPT-4o-mini / Haiku):
- 复盘需要语义理解 v0.3.1 人格 + 解读潜台词,thinking 模式更稳
- 多 provider 会增加配置 / 错误面 / 账户管理复杂度
- 当前 N=3 的频率已经把成本压到 1/3,跟便宜模型 + 每轮跑接近

**实测**:Kimi 一次分析 86s(reasoning + JSON 输出),后台跑用户不感知。

## 4. 4 层硬约束(PM 顾虑的"LLM 行为不稳定"对策)

### 4.1 JSON schema 校验(Pydantic)

```python
class TurnAnalysisResult(BaseModel):
    memories_to_add:           list[_MemoryToAdd]
    character_state_update:    _CharacterStateUpdate
    relationship_state_update: _RelationshipStateUpdate
    verifier_report:           str | None
```

校验规则:
- `delta` 在 [-10, +10] 区间(超出 → 整个 result 拒绝)
- `prompt_summary` ≤ 300 chars(模板说 ≤80 字,300 是硬上限)
- `memory.author` 必须是 "fact"(P5 不允许写 user_msg / ai_reply,那些路径已经在 chat 流程里写)
- `memory.content` 长度 ≥ 2 ≤ 500

### 4.2 stage 升级阶梯(`_validate_stage_transition`)

```python
_STAGE_UPGRADE_TRUST_THRESHOLD = {
    "stranger":  10,   # 升 known 需 trust >= 10
    "known":     25,   # 升 familiar
    "familiar":  40,   # 升 softened
    "softened":  55,   # 升 testing
    "testing":   65,   # 升 attached
    "attached":  75,   # 升 trusted
    "trusted":   85,   # 升 intimate
    "intimate":  100,  # 不能再升
}
```

规则:
- LLM 建议 stage 跳级(差 > 1)→ 拒绝(LLM 错位,保持原值)
- 升 1 但 trust 不够 → 拒绝
- 降 1 → **允许**(关系倒退合理,defense 涨太多说明用户冒犯了她)

### 4.3 Delta clamp 到 [0, 100]

`trust / intimacy / defense_level` 算 `current + delta` 后强制 clamp。
LLM 即使一时给 delta=+10,trust 从 95 也只能涨到 100,不会越界。

### 4.4 Per-user asyncio.Lock(防并发)

```python
_user_locks: dict[int, asyncio.Lock] = {}

if lock.locked():
    return False  # 直接 skip,不排队
async with lock:
    ...
```

**lock_held → skip,不排队**:如果用户连发 5 条消息,只让第一个 analyzer 跑,后续 4 个直接跳。
排队会导致 N 次 chat 后 N 个 analyzer 串行 86s × N 秒 → 状态滞后 + 资源占用。
跳过的好处:第一个 analyzer 跑完后 last_analyzed_message_id 推进,下次 chat 触发时阈值会自然重新满足。

## 5. analyzer prompt 设计

`prompts/turn_analyzer.md` 注入:
- `core_card`(小七人格,辅助理解对话语境)
- `character_state_block` / `relationship_state_block`(分析前的状态)
- 最近 N 条对话(用 "用户:" / "小七:" 而不是 user/assistant — 自然语言更易读)
- 4 个分析任务(memories / character / relationship / verifier)
- **严格 JSON 输出**,不要 markdown 包裹

关键约束(prompt 里直说):
- 大部分回合不该变化(null 是默认)
- mood/activity 只在对话里**确实暗示了变化**时给值
- delta 通常 0,只有"明显发生"时才动
- stage 升级**极其保守**,顶多升一级
- prompt_summary ≤ 80 字,**像她自己写的笔记**(避免机构腔)
- 不要捏造没在对话里出现的 fact

实测验证(见 §7):LLM 完全遵守这些约束 — 三个 delta 都是 ±1,没瞎升级,prompt_summary 第一人称写得自然。

## 6. 接入主链路

### 6.1 chat.py stream_reply finally 末尾

```python
# 不 shield(分析失败可丢,下次 last_analyzed 没动会重跑)
# 不 await(阻塞 SSE 收尾就违反 PM 不阻塞约束)
# cancelling 时跳过(客户端断了,这一轮算未完成)
if not cancelling:
    asyncio.create_task(
        analyze_turn_if_due(user_id, trace_id=trace_id),
        name=f"turn_analyzer-u{user_id}-{trace_id}",
    )
```

### 6.2 scheduler.py proactive_turn 末尾同模式

主动消息也算一段对话,该被复盘(可能用户的反应改变了 state)。
proactive 没 user 消息,所以 new_user_msg_count 通常不满阈值 → analyzer 自己 skip。
依然 spawn 是为了"用户接了几条 chat 之后,主动消息触发可能是合适的复盘时机"(让 last_analyzed_message_id 推进的机会)。

## 10. 已知未做(留给观察 / 后期)

### 10.1 无 model_run_logs 入库
analyzer 的 LLM 调用 / state 变化 / memories 写入只在 stdout log。
真出问题事后排查能 grep `trace=`,但批量分析(比如"哪种对话 analyzer 最常瞎升 stage")不方便。
P5.5 一并做(跟 chat.context_budget 一起入库)。

### 10.2 prompt_summary 累积式
当前 analyzer 每次会写一个新的 prompt_summary,**完全覆盖旧的**。
理想行为:增量摘要(把"上次的 summary + 这次的对话" → 新 summary)。
P5 当前的妥协:LLM 看到 character_state_block 里包含旧 summary,自然会综合写出新版。
如果发现"小七忘了几周前的事",P5.5 加专门的 summary 累积逻辑。

### 10.3 verifier_report 永远空
LLM 在我的 smoke 里输出了 `verifier_report=null`(对的 — 没异常)。
真正的 verifier(图风一致性 / 关系健康 / 越界检测)P3+ ImageVerifier 阶段才上,跟 P5 的 verifier_report 字段配套。

### 10.4 lock 不持久化
进程重启 → 锁丢。如果重启时正在跑 analyzer,新进程会重跑相同对话。
当前可接受(analyzer 幂等,重跑只是浪费 LLM tokens)。

### 10.5 elapsed 86s 偏长
Kimi K2.6 thinking 一次复盘 86s。如果用户操作密度高,lock_held 跳过会让 state 落后(批量更新)。
**这是设计选择(选 2.A 主模型)的代价**。如果 PM 觉得太慢,可以:
- 改用 Kimi 非 thinking 模式(改 model 参数,大概省一半时间)
- 改用 GPT-4o-mini(便宜 + 快,~5-10s)
- 都不用改 P5 主代码,只换 chat_provider 配置

## 12. 顺手发现

### 12.1 第一个 analyzer 实测 LLM JSON 输出非常规整
没有 markdown 包裹,字段全部正确,数值 ±1 全在范围。**Kimi K2.6 thinking 处理 JSON 输出任务很稳**。后续如果要换便宜模型,需要测它会不会输出 markdown / 越界 delta。

### 12.2 prompt_summary 写出口吻惊喜
LLM 输出 "他老让我提醒喝水又不喝。画了一张窗外给他。他说下周面试很紧张,我让他早点睡。" —— 真的像小七自己的笔记,而且把 4 段对话浓缩成 4 个事件。这是 prompt 里"像她自己写的笔记"约束起作用。

### 12.3 stage 阈值的合理性
当前阈值 trust ≥ 10 才能升 known。按每轮 +1~+3 的速度,**至少需要 5-10 轮真情实感的对话才会升级**。可能偏慢(用户聊了 1-2 周还是 stranger 显得呆)。
P5 跑一阵后再调阈值。低值方案(5/15/25/35/45/55/65)可以让阶梯更紧凑。
